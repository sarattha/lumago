Go + Vulkan 2D Game Engine Plan

A PC-first roadmap for building a custom 2D game engine in Go, with Vulkan used for normal-mapped lighting, per-light shadow maps, and experimental SDF-based soft shadows.

1. Summary

The engine should be designed as a Go-first 2D engine with a Vulkan renderer backend. Go should own game-facing concepts such as sprites, tilemaps, animation, scene data, transforms, lights, and occluders. Vulkan should own GPU resources, render targets, shader pipelines, command buffers, and compositing.

Core design rule: Go decides what should be rendered. Vulkan decides how it is rendered.
Area Recommended owner Reason
Sprites, animation, transforms Go engine These are gameplay and scene-level concepts.
Light objects and occluders Go engine The game world should define where lights and blockers exist.
Texture upload, GPU buffers, render passes Vulkan backend These are low-level rendering concerns.
Normal-map lighting, shadow maps, SDF raymarching Vulkan backend These are shader/GPU-heavy operations.
Render command generation Go engine The engine should batch high-level data before sending it to Vulkan.

Important performance rule: Do not call Vulkan once per sprite from Go. Generate arrays of render commands and submit them in batches. 
// Bad pattern:
for _, sprite := range sprites {
    vulkanDrawSprite(sprite) // too much per-sprite boundary overhead
}

// Better pattern:
commands := make([]SpriteDrawCommand, 0, len(sprites))

for _, sprite := range sprites {
    commands = append(commands, sprite.ToDrawCommand())
}

renderer.SubmitSprites(commands)

3. Lighting and Shadow Options

Option Feature MVP Priority Purpose
B Normal-mapped 2D lighting Core MVP Make sprites visually react to dynamic lights.
C Per-light 2D shadow maps Core MVP, simplified Allow lights to be blocked by walls, objects, and occluders.
D SDF-based soft shadows Experimental MVP / Phase 2 Support stylized soft shadows and raymarched light visibility.

Option B: Normal-Mapped 2D Lighting

This should be the first advanced visual feature. Each sprite may have an albedo texture and a normal map texture. The renderer writes albedo and normal data to separate render targets, then uses a light accumulation pass to calculate per-pixel lighting. 

Pass 1: Albedo pass
  Render sprite color into sceneColor.

Pass 2: Normal pass
  Render sprite normals into sceneNormal.

Pass 3: Light accumulation pass
  For each visible light, calculate lighting using sceneNormal.

Pass 4: Composite pass
  final = sceneColor * lighting

Light model

type Light2D struct {
    Position  Vec2
    Radius    float32
    Color     Color
    Intensity float32
    Falloff   float32
}

Material model

type Material2D struct {
    Albedo    TextureID
    Normal    TextureID
    Roughness float32

}

Option C: Per-Light 2D Shadow Maps

This is the second core feature. The Go engine owns occluder data, while Vulkan generates the shadow result per light. 

For each light:
  1. Gather nearby occluder segments.
  2. Convert occluder edges into light-relative space.
  3. Render or compute nearest blocking distance per angle.
  4. Store result in a 1D shadow texture or small 2D polar texture.
  5. During light rendering, compare pixel distance against blocker distance.

Occluder model

type Occluder2D struct {
    Points []Vec2
    Layer  int
}

Recommended MVP scope

  • Line segment occluders
  • Rectangle occluders
  • Tilemap wall occluders
  • Hard shadows first
  • Optional blur later

Option D: SDF-Based Shadows

SDF shadows should be treated as an experimental rendering path. They are powerful, but they are more complex than normal maps and standard shadow maps. 

For each pixel affected by light:
  ray = pixelPosition - lightPosition
  march along ray
  sample SDF
  if distance <= threshold:
      shadowed
  else:
      lit

Recommended MVP approach

Start with a static SDF texture generated from tilemap or level geometry. Upload it to the GPU, then raymarch in the fragment shader.

Recommendation: Keep Option C as the practical production shadow path and Option D as the advanced experimental soft-shadow path.

4. Phase Roadmap

Phase 0 — PC Vulkan Foundation Bootstrap

Goal: Create the minimum desktop runtime that opens a window and renders one textured quad through Vulkan.

Checklist

  • Create window
  • Create Vulkan instance
  • Create physical and logical device
  • Create swapchain
  • Create command buffers
  • Create synchronization objects
  • Render one textured quad
  • Capture frame in RenderDoc

Acceptance Criteria

  • A desktop window opens.
  • A textured quad is rendered.
  • The frame can be captured in RenderDoc.
  • The engine runs a fixed Update + Draw loop.
Phase 1 — Sprite Engine and Batching 2D Core

Goal: Let Go handle sprites, animation, transforms, camera, layer sorting, and batch command generation.

Checklist

  • Texture loader
  • Texture atlas
  • Sprite command buffer
  • Instance buffer
  • Layer sorting
  • Camera transform
  • Batch rendering

Acceptance Criteria

  • Render 1,000+ sprites.
  • Sort by layer and Z.
  • Use texture atlas.
  • Avoid per-frame heap allocation in the hot path.
  • Avoid per-sprite Vulkan calls.
Phase 2 — Normal-Mapped 2D Lighting Option B

Goal: Sprites react to dynamic lights using normal maps.

Checklist

  • Material supports albedo + normal textures
  • Scene color render target
  • Scene normal render target
  • Light buffer render target
  • Point light shader
  • Ambient light
  • Final composite pass

Acceptance Criteria

  • Sprites visually react to moving lights.
  • Normal maps work on characters, walls, floor, and props.
  • Multiple lights are supported.
  • Ambient light is configurable.
Phase 3 — Per-Light 2D Shadow Maps Option C

Goal: Lights are blocked by 2D occluders.

Checklist

  • Occluder component
  • Segment extraction
  • Nearby occluder culling
  • Per-light shadow texture
  • Shadow lookup in light shader
  • Hard shadows
  • Debug shadow visualization

Acceptance Criteria

  • Point lights cast visible shadows.
  • Rectangular wall occluders block light.
  • Characters or props can optionally cast shadows.
  • 3–8 dynamic shadow-casting lights are possible in a simple scene.
Phase 4 — SDF-Based Experimental Shadows Option D

Goal: Use signed distance fields to create soft, stylized 2D shadows.

Checklist

  • Generate static SDF from level geometry
  • Upload SDF texture
  • Raymarch from light to pixel
  • Render soft shadow factor
  • Compare with shadow-map mode
  • Debug SDF view

Acceptance Criteria

  • A static tilemap level can generate an SDF.
  • One light can use SDF raymarching.
  • Soft shadows are visible.
  • SDF renderer can be toggled on/off for comparison with shadow maps.
Phase 5 — Polish and Performance Stability

Goal: Make the MVP stable, measurable, and testable.

Checklist

  • Frame timing overlay
  • Draw call counter
  • Sprite counter
  • Light counter
  • GPU timing markers
  • Memory allocation checks
  • Shader reload in development mode
  • Basic config file

6. MVP Rendering Pipeline

Frame
  |
  |-- Go Update()
  |     |-- update sprites
  |     |-- update animations
  |     |-- update lights
  |     |-- update occluders
  |
  |-- Go BuildRenderCommands()
  |     |-- SpriteDrawCommand[]
  |     |-- Light2D[]
  |     |-- Occluder2D[]
  |
  |-- Vulkan Pass 1: Scene color
  |     |-- render albedo sprites
  |
  |-- Vulkan Pass 2: Scene normal
  |     |-- render normal-map sprites
  |
  |-- Vulkan Pass 3: Shadow maps
  |     |-- build per-light shadow maps from occluders
  |
  |-- Vulkan Pass 4: Light accumulation
  |     |-- render lights
  |     |-- sample normal texture
  |     |-- sample shadow map
  |
  |-- Vulkan Pass 5: SDF shadow mode, optional
  |     |-- sample SDF texture
  |     |-- raymarch light visibility
  |
  |-- Vulkan Pass 6: Composite
  |     |-- sceneColor * lightBuffer + emissive
  |
  |-- Present

Simple First Renderer Design

Pass 1: Sprite Color Pass
  Draw all visible sprites into sceneColor texture.

Pass 2: Sprite Normal Pass
  Draw all normal maps into sceneNormal texture.

Pass 3: Shadow Map Pass
  For shadow-casting lights, generate shadow data from occluders.

Pass 4: Light Accumulation Pass
  Calculate lighting using light data, normals, and shadow data.

Pass 5: Composite Pass
  final = sceneColor * lightBuffer + emissive
  render to swapchain image.

8. Example Engine API

Game code should stay simple. A developer should not need to call Vulkan directly. 

func main() {
    game := engine.NewGame(engine.Config{
        Width:  1280,
        Height: 720,
        Title:  "Go2D Lighting MVP",
    })

    scene := scene.New()

    floorMat := graphics.Material2D{
        Albedo: game.Assets.LoadTexture("floor.png"),
        Normal: game.Assets.LoadTexture("floor_n.png"),
    }

    scene.AddSprite(graphics.Sprite{
        Material: floorMat,
        Transform: graphics.Transform2D{
            Position: graphics.Vec2{X: 0, Y: 0},
            Scale:    graphics.Vec2{X: 1, Y: 1},
        },
    })

    scene.AddLight(graphics.Light2D{
        Position:    graphics.Vec2{X: 300, Y: 220},
        Radius:      400,
        Color:       graphics.Color{R: 1.0, G: 0.85, B: 0.55, A: 1.0},
        Intensity:   2.0,
        CastShadows: true,
    })

    scene.AddOccluder(graphics.Occluder2D{
        Points: []graphics.Vec2{
            {X: 200, Y: 200},
            {X: 400, Y: 200},
            {X: 400, Y: 260},
            {X: 200, Y: 260},
        },
    })

    game.SetScene(scene)
    game.Run()
}

10. Final Recommendation

This engine direction is technically sound for a PC-first 2D game engine. The cleanest plan is to build a Go-native engine API, keep game developers away from Vulkan details, and use Vulkan internally for a multi-pass 2D renderer. 

Best first target:
  Go engine core
  Vulkan desktop renderer
  Sprite batching
  Normal-mapped lighting
  Per-light hard shadows
  Optional SDF soft-shadow prototype

The most important engineering principle is to keep the renderer abstraction stable. If the engine API remains simple, you can later add alternative backends, improve shadow algorithms, or add mobile support without rewriting the game-facing layer.
