LumaGo Render Pipeline

A frame starts as simple Go scene data: sprites, materials, lights, occluders, and a camera. The renderer turns that data into batches, render targets, lighting buffers, and finally a swapchain image that is presented as the next output frame.

Go-native scene API Renderer interface boundary Vulkan-first backend 2D lighting and shadows

One Frame In Six Moves

1

Update Game

Your update function changes positions, animations, lights, camera, and gameplay state.

2

Build Scene Data

Scene exposes sprites, lighting config, lights, occluders, and camera.

3

Build Batch

Sprites become a SpriteBatch with vertices, indices, and draw stats.

4

Upload Frame

The backend uploads vertex, index, light, shadow, and SDF data for the frame.

5

Run Passes

Color, normal, shadow, light, optional SDF, and composite work produce the final image.

6

Present

The renderer submits commands, signals sync objects, and presents the swapchain image.

Code Path

The app loop owns frame timing and calls the renderer through the backend-neutral renderer.Renderer interface. Game code does not call Vulkan directly.

Game.runFrame()
  BeginFrame(scene.Camera())
  SubmitSpriteBatch(scene.BuildSpriteBatch(width, height))
  ConfigureLighting(scene.LightingConfig())
  SubmitLights(scene.Lights())
  SubmitOccluders(scene.Occluders())
  EndFrame()

What Crosses The Boundary

The renderer receives already-shaped engine data, not gameplay objects. This keeps Vulkan details behind the backend.

Renderer input:
  Camera2D
  SpriteBatch
  LightingConfig2D
  []Light2D
  []Occluder2D

Renderer output:
  FrameStats
  swapchain image

Pass Diagram

The composite formula for the full lighting path is sceneColor * lightBuffer + emissive. Debug views can swap the final output to show color, normals, light, shadow, or SDF data while developing.

Render Targets And Why They Exist

Buffer or Data
Written By
Used For
scene_color
Sprite color pass
Stores albedo color for each visible sprite pixel.
scene_emissive
Sprite color pass
Adds pixels that should glow without needing light.
scene_normal
Sprite normal pass
Stores normal-map direction so 2D sprites respond to light shape.
shadow data
Shadow map or SDF path
Stores light visibility from occluders before lighting is accumulated.
light_buffer
Light accumulation pass
Combines ambient light, point lights, falloff, normals, and shadow visibility.
swapchain image
Composite pass
The image sent to the desktop window by the present queue.
Game code owns intent

A user adds sprites, lights, occluders, and camera changes to scene.Scene. The game layer answers what should be drawn.

The scene builds renderable data

BuildSpriteBatch applies the camera and viewport, then creates batched geometry for the renderer.

The backend owns GPU details

The Vulkan renderer handles swapchain images, command buffers, buffers, descriptors, shader reloads, and presentation.

Current Backend Note

The active Vulkan backend is still evolving. It currently records a swapchain render pass and draws the prepared lit sprite batch, while exposing pass timings for color, normal, shadow, light, sdf, composite, and present. The full render-target graph shown above is the MVP lighting pipeline documented by the engine architecture.

How To Inspect A Frame

Run the lighting example and choose a debug view with environment variables. The overlay reports sprite count, light count, occluder count, draw calls, allocation bytes, CPU frame time, and pass timings.

make shaders
LUMAGO_DEBUG_VIEW=normal make run-lighting
LUMAGO_DEBUG_VIEW=light make run-lighting
LUMAGO_SHADOW_MODE=sdf make run-lighting

Mental Model For New Users

1. Author normal Go objects

Use graphics.SpriteDrawCommand, graphics.Material2D, graphics.Light2D, graphics.Occluder2D, and graphics.Camera2D.

2. Let the engine batch

Avoid thinking in per-draw Vulkan calls. The scene and renderer turn many sprites into compact frame data.

3. Debug by buffer

If lighting looks wrong, inspect color first, then normal, then light, then shadow or SDF, then final composite.
